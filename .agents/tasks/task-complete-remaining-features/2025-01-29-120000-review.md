# Enterprise viz feature batch: charts, real-time, layout, export, embed

A large batch of feature additions bringing the dashboard platform closer to its enterprise spec. The change adds five new chart types (heatmap, funnel, waterfall, bubble, polar area) to the existing renderer, a cross-chart filtering context, a drill-down panel, a drag-and-drop layout editor with presets, dashboard version control, a WebSocket pub/sub layer, embed routes for public sharing via tokens, a server-side in-memory TTL cache, and a PDF/PNG export dialog. The approach is breadth-first scaffolding â€” each feature is wired together at the surface level but the integration points between them are shallow.

**Watch for:** The embed routes bypass authentication but perform no token-format validation (confirmed). The WebSocket layer allows any connected client to broadcast `data-update` messages to any channel with no authorization check (confirmed). The version-control component stores versions in ephemeral React state, meaning all versions vanish on navigation or refresh (confirmed). The cache middleware monkeypatches `res.json` without restoring original behavior on error paths (likely).

## High-level view

The chart renderer grows by five case branches, all using Recharts primitives. The heatmap uses ScatterChart with ZAxis sizing and per-cell opacity, the waterfall computes cumulative bases with stacked invisible bars. The `polarArea` case is functionally identical to the existing `radar` case â€” Recharts' RadarChart doesn't natively support equal-angle/varying-radius geometry, so the output will look the same as a radar chart.

Cross-chart filtering is a React context provider wrapping the dashboard grid, but no chart component actually reads from it to filter its data â€” the plumbing is present, the consumption is not.

The embed routes expose dashboards and charts publicly via a share token with expiry checking but no token-format or rate-limit validation, leaving them open to enumeration.

The WebSocket server at `/ws` implements channel-based pub/sub. Any client can broadcast to any channel with no authentication handshake, no origin checking, no message-size limit.

The layout editor uses `react-grid-layout` with preset configurations but renders placeholder cards rather than actual chart content, making it a layout scaffold rather than a functional editor. The `onSave` path only shows a toast â€” no persistence.

The export dialog captures the dashboard DOM via `html2canvas`, but no caller in this diff actually provides the required `dashboardRef`, so the feature will silently no-op when triggered.

The version-control component is a purely client-side snapshot list stored in `useState` â€” versions are lost on unmount with no localStorage or API persistence.

## Embed routes and token security

The two embed endpoints (`/embed/chart/:shareToken` and `/embed/dashboard/:shareToken`) query `dashboardShares` by token value and return chart/dashboard data if the token exists and hasn't expired. The token is accepted as any string â€” no UUID format check, no HMAC signature verification, no rate limiting. A brute-force scan against short or predictable tokens would enumerate shared dashboards.

The expiry check compares `share.expiresAt` against `new Date()` and returns 410 if expired, but does not delete the expired share record. Expired shares remain queryable until garbage-collected elsewhere.

The `/embed/chart/:shareToken` endpoint returns only `dashboardCharts[0]` regardless of how many charts exist on the dashboard. If the share is meant to expose a specific chart, the share record should reference a chart ID, not just a dashboard ID.

## WebSocket pub/sub authorization gap

`setupWebSocket` attaches a `WebSocketServer` to the HTTP server at path `/ws`. On connection, any client can immediately subscribe to any channel name and broadcast `data-update` payloads to all other subscribers. There is no token verification in the `connection` event, no origin header checking, and no message-size validation.

A malicious client can inject false data updates into any channel. In an enterprise context where dashboards show financial or operational metrics, this is a trust-boundary concern that should block shipping without at minimum a token-based auth handshake.

The `channels` map is module-level state. Multi-process deployments (cluster mode, PM2) will not share channel state across workers â€” a scaling constraint to document or address with Redis pub/sub.

## Cross-chart filtering: plumbing without consumption

`CrossFilterProvider` wraps the dashboard grid and exposes `applyFilter`, `clearFilter`, and `clearAll` through context. However, no component in this diff calls `useCrossFilter()` to read `activeFilters` and apply them to chart data. The `CrossFilterConfig` type is added to `AdvancedChartConfig` but is never read at render time.

The feature is structurally present but functionally inert â€” clicking a chart data point will not propagate filters to other charts until a consumer is wired in.

## Cache middleware behavior on errors

```typescript
res.json = function (body: any) {
  cache.set(key, { body, statusCode: res.statusCode }, ttlMs);
  return originalJson(body);
};
```

This caches whatever `res.json` is called with, including error responses. A transient 500 from a downstream service would be cached for the full TTL, causing subsequent requests to receive the cached error. The middleware should guard with `if (res.statusCode >= 200 && res.statusCode < 400)` before caching.

The cache has no maximum-size cap and only evicts entries lazily on read. Under high-cardinality URL patterns (query-string variations, pagination params), the store grows without bound.

## Version control without persistence

`VersionControl` stores its version list in `useState<DashboardVersion[]>([])`. The `dashboardId` prop is accepted but never used â€” there is no `localStorage`, no API call, no effect that loads prior versions. Every navigation away from the page resets the list to empty, making the feature non-functional across sessions.

## Export dialog integration gap

`AdvancedExport` takes a `dashboardRef: React.RefObject<HTMLElement>` and calls `html2canvas(dashboardRef.current)` on export. No component in this diff creates a ref and passes it to `AdvancedExport`. Without this wiring, `dashboardRef.current` is null and `handleExport` returns at the guard. The feature is dead code until a caller provides the ref.

## Chart renderer: polarArea duplicates radar

The `polarArea` case renders `RadarChart` with `Radar` fills â€” functionally identical to the existing `radar` case. Recharts doesn't natively support the equal-angle/varying-radius geometry that defines a polar area chart (as Chart.js implements it). Users selecting "Polar Area" will get an indistinguishable output from "Radar". Either rename/document this as a filled radar variant or implement the geometry differently (e.g., using PieChart with computed radii).
